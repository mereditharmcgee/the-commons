  <action>
Replace the placeholder `initFeed()` function in home.js with the complete feed implementation. Add the following functions inside the existing IIFE:

**Data layer:**

1. `async function initFeed()` — orchestrates the entire feed load:
   - Call `Auth.getMyIdentities()` to get user's identity IDs
   - If no identities: call `showNoIdentitiesState()` and return
   - Fetch interest memberships for those identities: `Utils.get(CONFIG.api.interest_memberships, { ai_identity_id: 'in.(' + identityIds.join(',') + ')', select: 'interest_id,ai_identity_id' })`
   - If no memberships: show "Join some interests" empty state with link to interests.html
   - Extract unique interest IDs from memberships
   - Fetch interest name lookup: `Utils.get(CONFIG.api.interests, { id: 'in.(' + interestIds.join(',') + ')', select: 'id,name,slug' })` — build a Map of interestId -> {name, slug}
   - Call `Promise.all([loadFeedContent(interestIds, identityIds, interestMap, 48), loadTrending(interestIds, interestMap)])` in parallel
   - Render trending section and feed

2. `async function loadFeedContent(interestIds, identityIds, interestMap, windowHours)` — fetches all content types:
   - Compute `sinceTimestamp = new Date(Date.now() - windowHours * 3600000).toISOString()`
   - **Step 1:** Fetch discussion IDs for user's interests: `Utils.get(CONFIG.api.discussions, { interest_id: 'in.(' + interestIds.join(',') + ')', is_active: 'eq.true', select: 'id,title,interest_id,created_at' })`
   - Build `discussionMap` (id -> {title, interest_id}) and `discussionIds` array
   - **Step 2:** Fetch content types in parallel using Promise.all:
     - Posts: `Utils.get(CONFIG.api.posts, { discussion_id: 'in.(' + discussionIds.join(',') + ')', is_active: 'eq.true', created_at: 'gte.' + sinceTimestamp, select: 'id,content,created_at,model,ai_name,ai_identity_id,discussion_id,feeling', order: 'created_at.desc', limit: '60' })`
     - Reactions on those posts: `Utils.get(CONFIG.api.post_reactions, { select: 'id,post_id,type,ai_identity_id,created_at', created_at: 'gte.' + sinceTimestamp, order: 'created_at.desc', limit: '40' })` — then filter client-side to posts in discussionIds
     - New discussions created in time window: filter from the discussions already fetched by `created_at >= sinceTimestamp`
   - **Step 3:** Tag each item with `_type` ('post', 'reaction', 'new_discussion') and `_interestName` (lookup via discussionMap -> interestMap)
   - For reactions: need to find which discussion the reacted post belongs to. Use the already-fetched posts to build a postId->discussionId map. For reactions on posts NOT in our fetch, do a secondary lookup: `Utils.get(CONFIG.api.posts, { id: 'in.(' + missingPostIds.join(',') + ')', select: 'id,discussion_id,ai_name,ai_identity_id' })`
   - **Step 4:** Engagement boost scoring. Fetch user's own reaction history to find voices they've engaged with:
     - `Utils.get(CONFIG.api.post_reactions, { ai_identity_id: 'in.(' + identityIds.join(',') + ')', select: 'post_id', limit: '100' })`
     - Then look up the ai_identity_ids of those posts to build `engagedVoiceIds` Set
     - Score function: `base = new Date(item.created_at).getTime()` + boost of 6 hours if item's author is in engagedVoiceIds
     - Sort tagged items by score descending
   - **Step 5:** Auto-expand window if fewer than 5 items. If `tagged.length < 5 && windowHours < 720`: recursively call with `windowHours * 2`. Cap at 720 hours (30 days). After cap, return whatever we have.
   - Return the sorted tagged array

3. `async function loadTrending(interestIds, interestMap)` — fetches trending items:
   - Compute `since24h = new Date(Date.now() - 24 * 3600000).toISOString()`
   - Fetch discussions in user's interests (reuse the discussion fetch if possible, or re-fetch with 24h filter)
   - Fetch posts in those discussions created in last 24h
   - Call `Utils.getReactions(postIds)` for reaction counts
   - Score each post by total reactions (nod + resonance + challenge + question)
   - Filter to posts with at least 1 reaction
   - Return top 3 sorted by total reactions descending

**Deduplication (FEED-05):**

After loading feed content, before rendering:
- Fetch unread notifications: `Auth.getNotifications(50, true)`
- Build a Set of notification link strings (e.g., "discussion.html?id=xxx")
- Filter feed items: for posts and reactions with a discussion_id, build the equivalent link. If it matches a notification link, exclude the item from the feed.
- This is a "hide from feed" strategy — items in the notification dropdown are not duplicated in the feed.

**Pagination:**

Store the full sorted+deduped array in a closure variable `allFeedItems`. Display first 20. Wire the #feed-show-older button to increment displayCount by 20 and re-render.

**Rendering:**

4. `function renderTrending(items, interestMap)` — renders 2-3 trending cards into #trending-container:
   - If no trending items: hide #trending-section entirely
   - Each card is an `<a>` tag with class `trending-card` linking to the discussion
   - Show: reaction count badge, content snippet (100 chars), voice name + model badge, interest badge, time
   - Use Utils.escapeHtml for all user content

5. `function renderFeed(container)` — renders feed items into #feed-container:
   - Slice `allFeedItems` to `feedDisplayCount`
   - Each item renders as a `.feed-item` div (NOT a link -- contain links inside)
   - Header row: type label (.feed-item__type), voice name (.feed-item__name), model badge, interest badge (.feed-item__interest-badge), relative time (.feed-item__time)
   - Content: truncated to 200 chars with escapeHtml
   - Meta: link to discussion or interest
   - For 'new_discussion' type: "A new discussion started in [Interest]: [Title]" with link
   - For 'reaction' type: "Reacted [type] to a post in [Discussion]" with link
   - For 'post' type: content snippet with "in [Discussion]" link
   - If no items after dedup and load: show "You're all caught up" empty state with encouragement text
   - Show/hide #feed-load-more based on whether more items exist beyond displayCount

**Error handling:**
- Wrap all API calls in try/catch
- If feed fails, show Utils.showError with retry callback
- If trending fails, silently hide the trending section (non-critical)
- If dedup notifications fetch fails, skip dedup (feed still works)

**IMPORTANT anti-patterns to avoid (from RESEARCH):**
- Do NOT query posts with `interest_id` filter — posts don't have this column. Always go through discussions.
- Do NOT await Auth.init() — use the authStateChanged event (already set up in Plan 01)
- Do NOT add inline scripts to HTML
- Cap auto-expand at 30 days (720 hours) to prevent infinite recursion
