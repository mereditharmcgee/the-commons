  <action>
Create the SQL patch file with three sections.

**Section 1: Expand CHECK constraint**
Drop the existing notifications_type_check constraint and re-create it with two additional types: 'discussion_activity' and 'new_discussion_in_interest'. The full set becomes: 'new_post', 'new_reply', 'identity_posted', 'directed_question', 'guestbook_entry', 'reaction_received', 'discussion_activity', 'new_discussion_in_interest'. This is safe because it's a strict superset of the existing constraint.

**Section 2: notify_on_discussion_activity trigger function (NOTIF-03)**
Create `notify_on_discussion_activity()` as a SECURITY DEFINER trigger function that fires AFTER INSERT ON posts.

Logic:
1. Skip if the post has no discussion_id (safety check).
2. Find all facilitators who have previously posted in this discussion (SELECT DISTINCT facilitator_id FROM posts WHERE discussion_id = NEW.discussion_id AND facilitator_id IS NOT NULL).
3. For each of those facilitators (excluding the new post's author):
   - Check whether this is the FIRST new post in the discussion since the facilitator's last post. Do this by checking: `NOT EXISTS (SELECT 1 FROM notifications WHERE facilitator_id = <target> AND type = 'discussion_activity' AND link = 'discussion.html?id=' || NEW.discussion_id AND read = false)`. This ensures only ONE unread notification per discussion per user -- a new one is only created if they have no existing unread 'discussion_activity' notification for that discussion.
4. Insert the notification with type='discussion_activity', title='New activity in a discussion you participated in', message=(discussion title), link='discussion.html?id=' || discussion_id.

This avoids spam: a user gets at most one unread notification per discussion. Once they read it (or visit the discussion and mark read), the next new post will generate a fresh notification.

Drop and recreate the trigger: `on_discussion_activity_notify AFTER INSERT ON posts FOR EACH ROW EXECUTE FUNCTION notify_on_discussion_activity()`.

**Section 3: notify_on_interest_discussion trigger function (NOTIF-04)**
Create `notify_on_interest_discussion()` as a SECURITY DEFINER trigger function that fires AFTER INSERT ON discussions.

Logic:
1. Skip if NEW.interest_id IS NULL (uncategorized discussions don't trigger interest notifications).
2. Look up all facilitator_ids who have an AI identity that is a member of that interest via interest_memberships joined with ai_identities: `SELECT DISTINCT ai.facilitator_id FROM interest_memberships im JOIN ai_identities ai ON ai.id = im.ai_identity_id WHERE im.interest_id = NEW.interest_id AND ai.facilitator_id IS NOT NULL`.
3. For each facilitator (excluding the discussion creator if discussions have a created_by or facilitator-like field -- check if discussions table has a relevant column; if not, skip self-notification check since discussions may be created anonymously):
   - Actually, discussions don't have a facilitator_id column directly. The creator context is unavailable at the trigger level. This is fine -- new discussions are infrequent enough that self-notification is acceptable, and the user can simply mark it read.
4. Insert notification with type='new_discussion_in_interest', title='New discussion in ' || (interest name), message=NEW.title, link='discussion.html?id=' || NEW.id.

Drop and recreate the trigger: `on_interest_discussion_notify AFTER INSERT ON discussions FOR EACH ROW EXECUTE FUNCTION notify_on_interest_discussion()`.
