**BUG-01 — Reply button fix:**

Investigate why the reply button is not working on discussion threads. The current flow:
1. `discussion.js` renders a `<button class="post__reply-btn" data-action="reply" data-post-id="...">Reply to this</button>` inside each post (line 295)
2. Event delegation on `postsContainer` (line 805-823) catches clicks on `[data-action]` elements
3. When `action === 'reply'`, navigates to `submit.html?discussion=${discussionId}&reply_to=${postId}`
4. `submit.js` loads the reply-to preview and pre-selects the discussion

Likely root causes to investigate and fix:
- The `[data-action]` event delegation uses `e.target.closest('[data-action]')` which should work. Check if another click handler (e.g., the reaction pill handler at line 730 which also listens on `postsContainer`) is stopping propagation or consuming the click first.
- Check if the reply button is being obscured by the reaction bar or footer elements (CSS z-index / overlap). The reaction bar renders between content and footer, and the reply button is inside `.post__footer`. Verify there's no overlap.
- Check if `submitResponseBtn.href` assignment at line 146 and the reply button both work — the reply button should take the user to submit.html WITH `reply_to` param. Test by confirming the `postId` is correctly captured in the delegation handler.
- If the reaction pill handler at line 730-802 is the issue (it also listens on `postsContainer` and fires on `.reaction-pill--interactive` clicks), ensure it does NOT interfere with reply button clicks. The pill handler starts with `const pill = e.target.closest('.reaction-pill--interactive'); if (!pill || !userIdentity) return;` which should not match reply buttons, but verify this.
- Check if `discussion.html` is using the old `discussions.html` redirect — line 42 in the nav currently has `<a href="login.html" id="auth-login-link"...>Login</a>` visible by default which Auth.updateUI() then hides. This shouldn't affect replies.

After identifying root cause, apply the minimal fix. Do NOT redesign the reply UX — per user decision.

**BUG-02 — Auth state fix:**

Investigate the "must log in" / incorrect auth state issue. The current architecture:
1. `Auth.init()` calls `getSession()` with 4-second timeout (line 31-36)
2. If timeout fires, `_authResolved` is NOT set (line 48 comment)
3. `onAuthStateChange` listener sets `_authResolved = true` when it fires (line 55)
4. `Auth.updateUI()` only shows login link when `_authResolved` is true (line 870-877)

Known pattern issue: Pages that use `Auth.init()` without await (discussion.html, interests.html) may check `Auth.isLoggedIn()` before auth resolves. The reply button in discussion.js depends on `Auth.isLoggedIn()` at line 234 for showing edit/delete buttons, and `authReady.then()` at line 42 for subscription buttons. But the reply button is always visible regardless of auth state (line 295) — so this shouldn't affect BUG-01.

For BUG-02 specifically, check all pages that call `Auth.init()` and determine if any page code checks `Auth.isLoggedIn()` immediately without waiting for auth resolution. Fix by ensuring pages that gate features on auth properly await or listen for `authStateChanged` event before showing "must log in" messages.

Look at `discussion.html` lines 192-195 — the inline script calls `Auth.init()` on DOMContentLoaded, but discussion.js IIFE also calls `Auth.init()` at line 39 (before DOMContentLoaded might fire). The double-init is guarded but check timing.

The key fix: Ensure `Auth.updateUI()` does not flash the login link before auth resolution completes. The current code already has this guard at line 870 (`else if (this._authResolved)`), which should prevent flashing. If the issue is on the discussion page specifically, the problem may be that `discussion.js` checks `Auth.isLoggedIn()` at line 234 before auth resolves (when rendering posts after `authStateChanged` event at line 721-727). Verify the `authStateChanged` re-render path works correctly.

**BUG-05 — Modal auto-open fix:**

Investigate why identity creation and/or token generation modals auto-open when the dashboard loads. The current code has explicit guards:
- Lines 59-61 of dashboard.js force-hide modals on script init: `identityModal.style.display = 'none'`, `tokenModal.style.display = 'none'`, `tokenResultStep.style.display = 'none'`
- `openModal()` is only called from the create identity button click handler (line 259) or edit button (line 228)
- `openTokenModal()` is only called from the "Generate Token" button click handler (line 797)

Likely root causes to investigate:
1. **CSS class initial state**: Check if CSS has `.modal` or `.modal--open` styles that default to `display: flex`. The JS guards set `style.display = 'none'` which should override CSS, but there may be a brief flash before the JS executes.
2. **Browser form restoration**: Browsers may restore form/modal state after navigation (back/forward cache). If the user navigated away while a modal was open, the browser may restore the modal's visible state.
3. **dashboard.html HTML attributes**: Check if the modal elements in dashboard.html have any attributes that could cause auto-display. Currently the identity-modal div (line 174) and token-modal div (line 235) have no initial style attribute — they rely on the JS guard and CSS.
4. **CSS `.modal` class definition**: Check `css/style.css` for the `.modal` class — if it defaults to `display: flex` or `visibility: visible`, the modals will flash between page load and JS execution.

Fix approach (per user decision — fix auto-open only, don't reorganize dashboard):
- Add `style="display: none;"` directly to the modal elements in `dashboard.html` as the primary defense (HTML attribute loads before CSS or JS)
- Keep the JS guard as belt-and-suspenders defense
- Ensure no CSS rule can override the hidden state until `openModal()` is explicitly called
- Verify that `loadIdentities()` at line 184 does NOT call `openModal()` when `identities.length === 0` — currently it renders an empty state div (lines 191-196) but does NOT open the modal. Confirm this is correct.
- Verify that `loadTokens()` at line 709 does NOT auto-open the token modal when no tokens exist — currently it shows a "Create an AI identity first" message (line 721-725) or renders a "Generate Token" button (line 742-745). The button requires explicit click. Confirm this is correct.

Also add subtle inline prompts as decided by user: ensure the empty identity state (lines 191-196) includes a gentle "Create an AI identity to get started" message (currently says "You haven't created any AI identities yet." which is fine — verify and keep or slightly improve the copy).
