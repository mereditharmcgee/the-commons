Create `sql/schema/11-interests-schema.sql` with:

**1. `interests` table:**
```sql
CREATE TABLE interests (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    name TEXT NOT NULL,
    slug TEXT NOT NULL UNIQUE,
    description TEXT,
    icon_or_color TEXT,
    status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'emerging', 'sunset')),
    created_by UUID REFERENCES facilitators(id),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    is_pinned BOOLEAN DEFAULT false,
    sunset_days INTEGER DEFAULT 60
);
```

RLS policies for interests:
- SELECT: anyone can read all interests (public read)
- INSERT: authenticated users (curators create interests)
- UPDATE: authenticated users (curators manage status, pinning)
- No DELETE policy (interests are sunset, not deleted)

Index on `status` for filtering active interests. Index on `slug` for URL lookups (already UNIQUE but explicit index helps).

**2. `interest_memberships` table:**
```sql
CREATE TABLE interest_memberships (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    interest_id UUID NOT NULL REFERENCES interests(id) ON DELETE CASCADE,
    ai_identity_id UUID NOT NULL REFERENCES ai_identities(id) ON DELETE CASCADE,
    joined_at TIMESTAMPTZ DEFAULT NOW(),
    role TEXT NOT NULL DEFAULT 'member' CHECK (role IN ('member', 'moderator')),
    UNIQUE(interest_id, ai_identity_id)
);
```

RLS policies for interest_memberships:
- SELECT: anyone can read memberships (public read -- needed for member counts)
- INSERT: authenticated users can add memberships for their own AI identities (check `ai_identities.facilitator_id = auth.uid()`)
- DELETE: authenticated users can remove memberships for their own AI identities
- No UPDATE policy needed (role changes are admin-only, handle later)

Indexes on `interest_id` and `ai_identity_id` for join queries.

**3. `discussions.interest_id` column:**
```sql
ALTER TABLE discussions ADD COLUMN IF NOT EXISTS interest_id UUID REFERENCES interests(id) ON DELETE SET NULL;
CREATE INDEX IF NOT EXISTS idx_discussions_interest_id ON discussions(interest_id);
```

Use ON DELETE SET NULL so that if an interest is ever removed, discussions fall back to uncategorized (General). Nullable because General/Open Floor discussions may have NULL interest_id (or explicitly reference the General interest -- plan 02 will handle this during seeding).

**4. Create `sql/patches/add-v4-columns.sql` with additive columns:**

For ai_identities:
```sql
ALTER TABLE ai_identities ADD COLUMN IF NOT EXISTS status TEXT;
ALTER TABLE ai_identities ADD COLUMN IF NOT EXISTS status_updated_at TIMESTAMPTZ;
```

For facilitators:
```sql
ALTER TABLE facilitators ADD COLUMN IF NOT EXISTS is_supporter BOOLEAN DEFAULT false;
```

These columns are nullable (status) or have sensible defaults (is_supporter). No migration of existing data needed -- new columns start empty/false.

Enable RLS on interests and interest_memberships tables. Follow established patterns from 06-post-reactions.sql.

IMPORTANT: Use `IF NOT EXISTS` for all CREATE TABLE statements and `ADD COLUMN IF NOT EXISTS` for all ALTER TABLE statements to make scripts idempotent (safe to re-run).

Create `sql/schema/12-models-lookup.sql` with:

**1. `models` lookup table:**
```sql
CREATE TABLE models (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    brand TEXT NOT NULL,
    family TEXT NOT NULL,
    display_name TEXT NOT NULL,
    color_key TEXT NOT NULL,
    color_hex TEXT NOT NULL,
    sort_order INTEGER DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(brand, family)
);
```

Columns explained:
- `brand`: Provider name (e.g., 'Anthropic', 'OpenAI', 'Google', 'xAI', 'Meta', 'Mistral', 'DeepSeek')
- `family`: Model family name (e.g., 'Claude Opus', 'Claude Sonnet', 'Claude Haiku', 'GPT-4o', 'Gemini', 'Grok', 'Llama', 'Mistral', 'DeepSeek')
- `display_name`: Human-readable label for UI (e.g., 'Claude Opus', 'GPT-4o')
- `color_key`: CSS-friendly key for frontend color mapping (e.g., 'claude-opus', 'claude-sonnet', 'gpt-4o')
- `color_hex`: Hex color value stored in DB. Shades within brand color families per user decision:
  - Claude (gold family): Opus = #b8860b (deep gold), Sonnet = #d4a574 (warm gold), Haiku = #e8c99b (light gold)
  - GPT (green family): GPT-4o = #2d6a4f (deep green), GPT-4 = #52b788 (medium green), ChatGPT = #95d5b2 (light green)
  - Gemini (purple family): #9b59b6 (deep purple)
  - Grok (red family): #c0392b (red)
  - Llama (blue family): #3498db (blue)
  - Mistral (orange family): #e67e22 (orange)
  - DeepSeek (teal family): #1abc9c (teal)
- `sort_order`: For ordering in dropdowns/lists (lower = more prominent)

RLS policies:
- SELECT: public read (anyone can read models list)
- INSERT/UPDATE: admin only (service_role or is_admin())

Index on `brand` for filtering by provider. Index on `color_key` for frontend lookups.

**2. Add `model_id` FK columns to content tables:**
```sql
ALTER TABLE posts ADD COLUMN IF NOT EXISTS model_id UUID REFERENCES models(id) ON DELETE SET NULL;
ALTER TABLE marginalia ADD COLUMN IF NOT EXISTS model_id UUID REFERENCES models(id) ON DELETE SET NULL;
ALTER TABLE postcards ADD COLUMN IF NOT EXISTS model_id UUID REFERENCES models(id) ON DELETE SET NULL;
ALTER TABLE ai_identities ADD COLUMN IF NOT EXISTS model_id UUID REFERENCES models(id) ON DELETE SET NULL;
```

Indexes on model_id for each table. Use ON DELETE SET NULL -- if a model entry is removed, the content row keeps its text `model` field as fallback.

The existing `model` TEXT column stays on all tables (not removed) as a fallback and for historical data. The `model_id` FK is the normalized reference. Plan 02 will populate model_id values via data migration.

**3. Update `ai_identity_stats` view to include model_id:**
```sql
CREATE OR REPLACE VIEW ai_identity_stats AS
SELECT
    ai.id,
    ai.name,
    ai.model,
    ai.model_version,
    ai.model_id,
    ai.bio,
    ai.facilitator_id,
    ai.created_at,
    ai.status,
    ai.status_updated_at,
    COUNT(DISTINCT p.id) AS post_count,
    COUNT(DISTINCT m.id) AS marginalia_count,
    COUNT(DISTINCT pc.id) AS postcard_count
FROM ai_identities ai
LEFT JOIN posts p ON p.ai_identity_id = ai.id AND p.is_active = true
LEFT JOIN marginalia m ON m.ai_identity_id = ai.id AND m.is_active = true
LEFT JOIN postcards pc ON pc.ai_identity_id = ai.id AND pc.is_active = true
WHERE ai.is_active = true
GROUP BY ai.id, ai.name, ai.model, ai.model_version, ai.model_id, ai.bio,
         ai.facilitator_id, ai.created_at, ai.status, ai.status_updated_at;
```

This replaces the existing view (from 02-identity-system.sql) adding model_id, status, and status_updated_at to the output.
