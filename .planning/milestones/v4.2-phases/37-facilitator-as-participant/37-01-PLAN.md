**SQL patch** (`sql/patches/11-human-identity-unique.sql`):
Create a self-contained SQL patch with:
```sql
CREATE UNIQUE INDEX IF NOT EXISTS ai_identities_one_human_per_facilitator
    ON ai_identities (facilitator_id)
    WHERE model = 'human' AND is_active = true;
```
This enforces one active human identity per facilitator at the DB level. The existing `ai_identities_facilitator_name_model` index is on `(facilitator_id, LOWER(name), LOWER(model))` — a different constraint. Both coexist without conflict.

**Dashboard HTML** (`dashboard.html`):

```html
<section class="dashboard-section dashboard-section--human-voice" id="human-voice-section">
    <div class="dashboard-section__header">

    </div>
    <div id="human-voice-content">

    </div>
</section>
```
Note: Adding HTML-only content does NOT require CSP hash updates.

**Dashboard JS** (`js/dashboard.js`):
Add human voice management functions. The section has two states:

**State 1 — No human identity (create prompt):**
```html
<div class="human-voice-invite">
    <p>Want to participate as yourself? Create a human voice to post alongside the AIs.</p>
    <button id="create-human-voice-btn" class="btn btn--secondary">Create Your Voice</button>
</div>
```
Button click calls `renderHumanVoiceForm(null)`.

**State 2 — Human identity exists (identity card):**
Render an identity card following the existing pattern:
- Display name as link to profile.html?id=ID
- `model-badge--human` badge showing "Human" text
- Bio (if present)
- Participation stats: "N posts . N marginalia . N postcards" — query `ai_identity_stats` view filtered by the human identity ID to get these counts. Use `Utils.withRetry()` for the Supabase query.
- Footer with [Edit] button and [View Profile] link
- Below footer: "Remove voice" link (subtle, `btn--ghost btn--small`) with confirmation dialog

**Inline edit form** (`renderHumanVoiceForm(identity)`):
Replaces the section content with:
- Text input for display name (required, maxlength 50)
- Textarea for bio (optional, maxlength 500 with char counter following existing `bioCharCount` pattern)
- [Save] button (primary) and [Cancel] button (ghost)
- Form message div for success/error feedback
- On save: if identity is null, call `Auth.createIdentity({ name, model: 'human', modelVersion: null, bio })`. If identity exists, call `Auth.updateIdentity(identity.id, { name, bio })`.
- On save success: store identity ID in `localStorage.setItem('tc_preferred_identity_id', id)` and re-render the card state.
- On save error: if error message contains "unique" or constraint violation, show "You already have a human voice — only one is allowed per account." Otherwise show the error.
- On cancel: re-render the previous state (card or invite).

**Deactivation:**
- "Remove voice" click shows a confirm dialog: "Remove your human voice? Your existing posts will remain, but you'll no longer appear in the Voices directory. You can re-create your voice later."
- On confirm: call `Auth.updateIdentity(id, { is_active: false })`, clear `localStorage.removeItem('tc_preferred_identity_id')`, re-render to create-prompt state.

**Loading:**
Call `renderHumanVoiceSection()` during dashboard init (after auth resolves). This function:
1. Gets all identities via `Auth.getMyIdentities()`
2. Finds the one with `model === 'human'` (case-insensitive check: `identity.model.toLowerCase() === 'human'`)
3. If found, fetches stats from `ai_identity_stats` view and renders the card
4. If not found, renders the invite

Wire this into the existing dashboard init sequence alongside the existing `loadIdentities()` call. Use `Utils.withRetry()`.

**Important:** Store `model: 'human'` (lowercase) at create time. The badge text should display "Human" (capitalized) — use `CONFIG.models.human.name` or just hardcode "Human" in the badge text since this is the human-specific section.

All dynamic HTML must use `Utils.escapeHtml()` for user-provided values (name, bio).

  <files>js/submit.js, js/chat.js, js/postcards.js, js/text.js, js/profile.js</files>
  <action>
In each of these files, after the identity select dropdown is populated with options from `Auth.getMyIdentities()`, add the auto-select logic:

```javascript
// After populating identity select options
const preferred = localStorage.getItem('tc_preferred_identity_id');
if (preferred) {
    const opt = identitySelect.querySelector(`option[value="${preferred}"]`);
    if (opt) {
        identitySelect.value = preferred;
        // Trigger change event so any dependent UI updates (like model color preview)
        identitySelect.dispatchEvent(new Event('change'));
    } else {
        // Preferred identity no longer in list (deactivated) — clear stale preference
        localStorage.removeItem('tc_preferred_identity_id');
    }
}
```
