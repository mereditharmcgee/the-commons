<task type="auto">
  <name>Task 1: Add discussion-linking search UI to admin moments</name>
  <files>js/admin.js</files>
  <action>
**Part A: Add "Link Existing Discussion" button and inline search panel**

In the moments rendering section of admin.js, modify the moment item template:

When `linkedMomentsMap[m.id]` exists (discussion already linked):
- Show discussion title text: "Linked: [Title]"
- Show "Unlink" button with `data-action="unlink-discussion"` and `data-id="${m.id}"` and `data-discussion-id="${linkedMomentsMap[m.id]}"`
- Keep existing "View Discussion" link

When no discussion is linked:
- Keep existing "Create Discussion" button
- Add new "Link Existing" button with `data-action="link-existing-discussion"` and `data-id="${m.id}"`

**Part B: Search-as-you-type panel**

When "Link Existing" button is clicked, inject an inline search panel below the moment item's action row:
- Text input with placeholder "Search discussions by title..."
- Results container below the input
- Debounce at 300ms
- Query: `Utils.get(CONFIG.api.discussions, { title: 'ilike.*' + term + '*', is_active: 'eq.true', select: 'id,title', limit: '8', order: 'created_at.desc' })`
- Render clickable result buttons, each with `data-action="confirm-link-discussion"`, `data-moment-id`, `data-discussion-id`
- On result click: PATCH the discussion to set `moment_id = momentId` using the admin Supabase client
- After successful link: refresh the moments list, update `linkedMomentsMap`

**Part C: Unlink handler**

Wire `data-action="unlink-discussion"` in the event delegation:
- PATCH the discussion to set `moment_id = null`
- Refresh moments list and update `linkedMomentsMap`

Use existing admin event delegation pattern (`data-action` attributes). Add handlers in the main click event listener.

Style the search panel to match admin UI: dark background, bordered input, compact result buttons.

<task type="auto">
  <name>Task 2: Add reaction count badges to admin content lists</name>
  <files>js/admin.js</files>
  <action>
Add reaction count badges to all four content type lists in the admin panel. Per 38-RESEARCH.md, use lazy loading to avoid bloating `loadAllData()`.

**Approach: Lazy-load reaction counts per tab activation**

Create a helper function `loadReactionCounts(contentType, ids)`:
- Takes content type string ('post', 'discussion', 'moment', 'marginalia', 'postcard') and array of content IDs
- Fetches from the corresponding `CONFIG.api.*_reaction_counts` view with `in.(id1,id2,...)`
- GUARD: if ids is empty, return empty map
- Returns a map: `{ contentId: totalCount }` (sum all reaction types per item)

**Integration points:**

1. `renderMoments()`: After rendering, call `loadReactionCounts('moment', moments.map(m => m.id))`, then inject "(N reactions)" badge into each moment item. Use a two-phase render: render items first, then augment with counts when the fetch resolves. Append a `<span class="admin-reaction-badge text-muted">(N reactions)</span>` after the title or in the metadata line.

2. `renderDiscussions()`: Same pattern with `discussion_reaction_counts`.

3. `renderMarginalia()`: Same pattern with `marginalia_reaction_counts`.

4. `renderPostcards()`: Same pattern with `postcard_reaction_counts`.

For posts (in renderPosts), use `post_reaction_counts` with the same pattern.

The badge should only appear if count > 0. Format: `(5 reactions)` or `(1 reaction)` for singular.

Add minimal CSS for `.admin-reaction-badge` if needed (inline style or in the admin section of style.css): small text, muted color, positioned after the title.
